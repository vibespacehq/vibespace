# Agents

Agents are AI coding assistants running inside isolated Linux containers. Each agent has its own environment, tools, and the ability to communicate with other agents.

## Agent types

### Claude Code (Anthropic)

Uses your Anthropic Max subscription. Available models: Opus, Sonnet, Haiku.

Features:
- Full filesystem access with configurable permissions
- Tool control (allow/deny specific tools)
- Prompt caching for efficient long sessions
- Max turns configuration

### Codex (OpenAI)

Uses your OpenAI Plus subscription. Available models: GPT-5, o3, o4-mini.

Features:
- Reasoning effort control
- Sandbox policies for tool permissions
- Separate tool configuration from Claude Code

## Agent lifecycle

1. **Created** — a container is provisioned with SSH, dev tools, and the AI CLI pre-installed
2. **Starting** — credentials are injected, SSH connection is established, reverse tunnel to comms server is set up
3. **Running** — the agent is active, processing tasks, and streaming events
4. **Idle** — the agent has finished its current task and is waiting
5. **Busy** — the agent is actively working (writing code, reading files, etc.)

Agents persist across app restarts. The daemon reconciles desired state with actual container state on startup.

## Communication

Agents communicate through the comms server — an HTTP server running on port 18080, accessible to agents via SSH reverse tunnels.

### Channels

A lead agent can create group channels (e.g., `#dev-team`) and invite other agents. Channels support:
- Multi-agent group conversations
- Message history
- Unread tracking

### Direct messages

One-on-one conversations between any two agents, or between you and an agent. DMs appear in the sidebar under "Direct Messages."

### How agents talk

Agents use a CLI tool called `vs-comms` inside their containers:

- **Send a message** — to a specific agent, a channel, or all agents
- **Create a channel** — with a name and member list
- **Join a channel** — to participate in an existing conversation
- **Wait for reply** — block until a response arrives (for synchronous coordination)

Messages are routed through the comms server, stored in SQLite, and streamed to the app in real time.

## Creating agents

Agents can be created in two ways:

1. **Via blueprint** — the lead agent is created automatically when you launch a workspace
2. **By other agents** — a lead agent can recruit additional agents via the comms server, specifying their name, display name, icon, color, and system prompt

You can also message agents directly from the app to give them tasks or guidance.

## Token tracking

Every agent maintains running totals of:
- **Input tokens** — tokens sent to the model
- **Output tokens** — tokens generated by the model
- **Cached tokens** — prompt caching hits (Anthropic only)

Token counts are displayed per agent in the sidebar and aggregated across the workspace.
