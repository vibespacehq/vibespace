[FIXED] BUG 11: Agent state lost after config change (pod restart)
----------------------------------------------------------------------
Location: Config persistence across pod restarts
Steps to reproduce:
  1. `vibespace create test-persist`
  2. `vibespace test-persist connect claude-1` - login/authenticate
  3. `vibespace test-persist config set claude-1 --allowed-tools "Bash,Read,Write,Edit"`
  4. Wait for pod restart
  5. `vibespace test-persist connect claude-1` - requires login again!

Expected: Agent state should persist across pod restarts
Actual: User must re-authenticate after any config change

Severity: CRITICAL
Impact:
  - Any config change logs out the user
  - Claude sessions/conversation history likely also lost
  - Makes config changes extremely disruptive
  - Users lose all work context

Root cause:
  - Home directory (~/) is ephemeral container storage
  - Only /vibespace is mounted on PVC
  - ~/.claude (credentials + sessions) lives in ephemeral storage
  - Pod restart = new container = everything in ~/ gone

Workaround:
  - Use `--share-credentials` flag - credentials persist BUT are shared across all agents
  - Not ideal if you want isolated credentials per agent

Current behavior:
  - Without --share-credentials: per-agent credentials, but ephemeral (lost on restart)
  - With --share-credentials: persistent credentials, but shared across all agents
  - No option for persistent AND isolated credentials

Fix needed:
  - Each agent should have persistent credentials stored on PVC
  - e.g., /vibespace/.agents/claude-1/.claude, /vibespace/.agents/claude-2/.claude
  - Symlink ~/.claude to agent-specific PVC path

BUG 14: Codex agents don't resume sessions - no conversation memory
-------------------------------------------------------------------
Location: TUI session handling for Codex
Evidence:
  [You → all] what is 2+2?
  [codex-1] 4
  [claude-1] 4
  [You → all] what was my previous message?
  [codex-1] Your previous message was: "what was my previous message?"  <- NO MEMORY
  [claude-1] Your previous message was "hi".  <- REMEMBERS
  [You → all] didn't i ask an addition question?
  [codex-1] I'm not sure what you're referring to  <- NO MEMORY
  [claude-1] Yes, you're right. Your previous message was "what is 2+2?"  <- REMEMBERS

Codex treats each message as a new conversation, Claude maintains context.

Severity: CRITICAL
Impact: Codex is unusable for multi-turn conversations in TUI

Likely cause in pkg/agent/codex/codex.go:
  - SessionIDFlag() returns "" (empty) - Codex auto-generates session IDs
  - ResumeFlag() returns "resume" - but may not be applied correctly
  - TUI session manager may not be handling Codex's session format properly

Fix needed:
  - Verify Codex CLI's actual session/resume mechanism
  - Ensure TUI correctly resumes Codex sessions with "codex exec resume <id>"

Related: /session commands also broken for Codex
  - /session @codex-1 new - may create session but won't be resumed
  - /session @codex-1 list - shows sessions that can't be resumed
  - /session @codex-1 resume <id> - won't work if resume is broken
  - /session @codex-1 info - meaningless without working sessions
